HOW IT WORKS

進め方

①教材を読む → ②実践課題(実技)を全部やってチェック → ③確認テストで全問正解 → 知識と実技の両方そろって合格、次のレッスンが解放される。
合格した問題はホームの「復習クイズ」にランダムで出てくる。反復して満点を維持できてこそ定着や。分からん用語はレッスン内のオレンジのチップか用語集タブで調べる。それでも詰まったら質疑提出シート(テンプレタブ)で真殿さんへ。

FORBIDDEN

三大禁止事項

1

指示なしで git push しない。例外なし。KYOEIでは push した瞬間に本番(お客様が使っている環境)へ反映される。push の前に必ず真殿さんの明示的な指示を確認する。

2

mainブランチで直接作業しない。作業は必ず自分のブランチ(feature/〜)を切ってから。

3

--force系・履歴改変コマンドを使わない。push --force / reset --hard(共有ブランチ)/ rebase(共有ブランチ)は真殿さん同席時のみ。

CRAFTFILE RULES

開発ルール 7ヶ条

1

git push は指示があるまで絶対しない(禁止事項の再掲。それだけ大事)

2

すべてのチェックポイントで next build がグリーンで通ること

3

スキーマ変更は冪等なSQLマイグレーションファイルで行う(直接ALTERしない)

4

新規Prismaモデルには必ず tenantId(付け忘れ=他社データ混在事故)

5

高さ指定は 100dvh 系を使う(100vh 禁止。モバイル対応の全社方針)

6

モーダルは共通の ModalShell を使う。独自モーダルを生やさない

7

タップターゲットは44px以上。スマホで押しやすいことを実機で確認

QUESTION FLOW

質疑ルール(AI資料化方式)

①まず15分は自力で調べる(AIに状況・やったこと・エラー全文・期待する結果をセットで渡す)→ ②解決しなければAIで質疑提出シートを作成(テンプレタブ)→ ③真殿さんに提出 → ④回答をナレッジベースに反映。
例外(即・直接声かけOK):本番影響のある操作の実行判断/お客様・納期の緊急事態/体調・メンタルの相談。
